Data Flow — What happens on each arrow

①
Developer → GitHub  (git push) Claude Code edits index.html locally, then commits and pushes to the GitHub repository using the st-andrews-academy account.
②
GitHub → GitHub Pages  (auto-deploy) GitHub detects a new commit on main and automatically rebuilds the Pages site. Takes ~2–5 minutes. No manual step needed.
③
GitHub Pages → Browser  (page load) When any user opens the URL, GitHub Pages serves the single index.html file (including all embedded images, CSS, and JavaScript) over HTTPS.
④
Firestore → Browser  (read on startup) On every page load, loadFromFirestore() fetches 15 Firestore docs in parallel: alumni, faculty, events, config, announcements, approved feedback, and all 9 pending queues (adds + edits for alumni/faculty/events/feedback, plus self-registrations). This is the live shared data — the same data every user sees. If Firestore is unreachable, the browser falls back to its localStorage cache.
⑤
Browser → Firestore  (write on every change) When a user adds, edits, or deletes records, the change is written to Firestore via saveDB(), saveFacDB(), savePending(), or individual event writes. Firestore failures are non-fatal — a console.warn is logged and the local change still persists.
⑥
Browser ↔ localStorage  (cache) All Firestore reads are mirrored to localStorage as a local cache. All writes also go to localStorage in parallel with Firestore. If Firestore is down, the app reads the last known state from this cache instead of the seed data.

Step-by-step Sequences

When a user opens the site

1
Browser downloads index.html from GitHub Pages CDN (includes all CSS, JS, and embedded images)
2
loadFromFirestore() runs — fetches 15 Firestore docs in parallel: alumni, faculty, events, config, announcements, approved feedback, and all 9 pending queues (adds + edits for alumni/faculty/events/feedback, plus self-registrations)
3
First-ever install only: if Firestore has no data, seeds it from ALUMNI_SEED / FAC_SEED and saves to Firestore for all future users
4
Firestore data is mirrored into localStorage as a local cache
5
Portal fully loads — all tabs, filters, and tables render. Loading overlay is hidden.
6
Firestore offline: falls back to localStorage cache. If cache is also empty, seeds from ALUMNI_SEED.

When a user saves a change

1
User adds/edits alumni, faculty, event, or approves registration
2
PIN modal confirms identity for protected actions (add, edit, delete, approve, restore, change PIN)
3
saveDB() / saveFacDB() / savePending() write the updated array to localStorage instantly
4
Same functions fire a Firestore write in parallel — change becomes visible to all users on their next page load
5
Toast notification: "Saved!" — user sees no delay. Firestore failure is logged silently, local change is preserved.

When you update the app

1
Claude Code edits index.html on developer workstation
2
git commit + git push to GitHub repository
3
GitHub automatically triggers a Pages deployment
4
~2–5 minutes later, new version is live globally
5
Firestore data is untouched — all alumni records, events, and faculty survive the update. Seed data changes only affect a brand-new Firestore instance.
6
Users refresh their browser to get the new app version, and Firestore loads their existing data as normal

PIN Access Control — Who Can Do What

Adding and editing records (alumni, faculty, events) is open to anyone — submissions go to a pending queue, not the live database. The Admin PIN gates approvals, deletions, restores, backups, exports, and settings saves. This mirrors the self-registration workflow: anyone can submit, only the admin can publish, export, or apply changes.

How the PIN works: The PIN is stored in saa_admin_pin in localStorage and synced to portal/config in Firestore. On startup, the Firestore PIN overwrites the local one. Falls back to default aneza654457 if absent in both. When a PIN-protected action is triggered, a modal appears asking for the PIN. On success, the action proceeds and is written to both Firestore and localStorage.

Pending queue flow: Add forms (alumni, faculty, event, feedback) and Edit forms all write to pending queues — 9 queues total across portal/pending_alumni, portal/pending_edit_alumni, portal/pending_faculty, portal/pending_edit_faculty, portal/pending_events, portal/pending_edit_events, portal/pending_feedback, portal/pending_edit_feedback, and portal/pending (self-reg). Announcements bypass the pending queue — admin writes them directly (Notices tab, top panel). The Settings tab shows all nine queues with an aggregate badge count. Admin approves (PIN) → record moves to/updates the live database. Admin rejects (PIN) → record is discarded.

Backup & Export: All backup downloads (alumni, faculty, Memory Lane, settings JSON) and CSV exports now require the admin PIN. Restoring is also PIN-protected because it overwrites stored data. Alumni restore (restoreData()) and Faculty restore (restoreFaculty()) are independent — each only overwrites its own dataset.

Architecture Comparison — SAA Alumni Portal vs CapulongFarms Online Orders

The Alumni Portal uses Firebase Firestore (Google-managed, no custom server). CapulongFarms Online Orders requires a Railway server because of payment processing, secret keys, and untrusted customers.

✅ Firebase is enough — St. Andrews Alumni Portal

👥Data is managed by trusted admins — alumni can't manipulate records
🔓Everyone is allowed to see everything — no private data per user
💸No money involved — no payment keys, no financial transactions
📭No webhooks needed — no external service calls back to the app
🧮Simple logic — add, edit, delete records. Nothing to compute server-side.
🔄Firestore handles cross-device sync — changes on one device are immediately visible to all others

❌ Firebase alone is not enough — CapulongFarms Online Orders

🔑PayMongo secret key must be hidden on a server — putting it in the browser exposes it to anyone who opens DevTools
📡PayMongo sends webhooks (payment confirmations) to a server URL — Firebase has no URL to receive them securely
📦Inventory must be validated server-side — if 10 people buy the last item, only a server can decide who gets it
🧾Customers must not edit their own orders — price, quantity, payment status must be locked by a trusted server
🧮Business logic is complex — discounts, tax, shipping, order status transitions need server-side rules
📧Email/SMS notifications on order events require a running server process
Question St. Andrews Alumni Portal CapulongFarms Online Orders
Secret keys needed? No Yes — PayMongo secret key must stay on the server
Payment processing? No Yes — PayMongo charges real money
Webhooks from outside? No Yes — PayMongo calls back to confirm payment success
Who writes data? Anyone can submit additions or edits (→ pending queue); admin approves/deletes (PIN-protected) Untrusted customers — must be validated server-side

Bottom line The St. Andrews Alumni Portal uses Firebase Firestore as its live database — changes are shared across all devices automatically at no cost. No custom server is needed because there are no secret keys to protect, no payments to process, and data is managed only by trusted admins. For CapulongFarms Online Orders, Railway is not optional: it is the security wall between customers and your money, your inventory, and PayMongo.
